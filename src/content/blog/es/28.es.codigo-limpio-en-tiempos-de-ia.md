## Hipótesis provocadora

Te propongo una pregunta incómoda:

> Si no vamos a leer el código, ¿a quién le importa que esté limpio?

Con agentes generando y modificando ficheros de forma continua, parece razonable pensar que el "código bonito" pierde valor.  
Si la máquina escribe para la máquina, podríamos aceptar cualquier cosa mientras funcione.

La tesis suena eficiente.  
También es incompleta.

---

## El contraargumento que cambia el marco

Aunque no lo lea un humano en cada iteración, el código sigue siendo leído una y otra vez.

Y ese lector no es uno, son muchos:

- el agente que implementa una feature,
- el agente que corrige un bug,
- el agente que refactoriza,
- el agente que genera tests,
- y tu yo del futuro cuando algo falle en producción.

La diferencia es que antes optimizábamos legibilidad para personas, y ahora debemos optimizarla también para agentes.

> El código limpio deja de ser una cuestión estética y pasa a ser una cuestión de rendimiento cognitivo del sistema completo.

---

## Los agentes son buenos con texto, no con barro

Los modelos son especialmente buenos interpretando texto claro, consistente y con intención explícita.  
Cuando el código está mezclado, ambiguo o lleno de ruido:

- empeora la precisión de sus cambios,
- aumenta la probabilidad de efectos colaterales,
- y sube el coste de cada ciclo de corrección.

Código limpio, en este contexto, significa texto técnico claro para quien va a operar sobre él: humano o agente.

---

## El error habitual: exigir limpieza total demasiado pronto

Aquí viene el matiz importante: no conviene saturar al agente con una batería infinita de reglas de "clean code" durante la fase de programación inicial.

Si le impones demasiadas restricciones desde el primer prompt:

- baja la velocidad de entrega,
- bloqueas exploración técnica,
- y terminas optimizando forma antes que comportamiento.

En etapas tempranas, manda la tracción.

> Primero algo sucio que funcione. Luego algo verificado. Después algo limpio.

---

## Flujo recomendado en 3 fases

### 1) Hacerlo funcionar (aunque esté sucio)

Objetivo: validar que la solución resuelve el problema real.

Reglas prácticas:

- limitarse a arquitectura mínima viable,
- evitar sobrediseño,
- priorizar ciclos cortos de implementación.

### 2) Blindarlo con tests

Objetivo: asegurar comportamiento antes de tocar estructura.

Reglas prácticas:

- cubrir casos críticos y bordes relevantes,
- fijar contratos públicos,
- usar los tests como red de seguridad para el refactor.

### 3) Limpiarlo con seguridad

Objetivo: mejorar legibilidad, modularidad y mantenibilidad sin romper funcionalidad.

Reglas prácticas:

- simplificar nombres y responsabilidades,
- eliminar duplicidad accidental,
- hacer refactor pequeño + test verde en cada paso.

---

## Orden de herramientas para la fase de limpieza

No todo debe resolverse con prompts sofisticados desde el minuto uno.  
Para limpiar con eficiencia en tiempos de IA, conviene un orden operativo:

### Paso A: herramientas deterministas locales primero

Empieza por lo mecánico y repetible:

- formateadores,
- linters,
- analizadores estáticos,
- reglas de estilo y complejidad.

Estas herramientas son baratas, previsibles y no "alucinan".

Si en tu stack aplica, aquí entran comandos de toda la vida:

- `npm run lint`
- `npm run test`
- `make test`
- `cargo clippy`

### Paso B: skills especializadas después

Cuando lo básico ya está estable, entonces sí:

- usa skills para refactors guiados,
- usa skills para detección de code smells complejos,
- usa skills para estandarizar patrones de equipo.

La skill no sustituye la base determinista: la potencia.

---

## Criterio operativo para equipos AI-first

Si trabajas con agentes a diario, este criterio suele funcionar bien:

1. **Rapidez primero** para validar valor.
2. **Tests después** para fijar comportamiento.
3. **Limpieza al final** para escalar sin deuda invisible.

Este orden no elimina el código limpio; lo pone en el momento en que realmente maximiza ROI.

---

## Conclusión

La pregunta inicial sigue siendo útil porque obliga a pensar:

> "Si no lo leerá un humano, ¿importa que esté limpio?"

Sí, importa.  
En un entorno de agentes, el código se convierte en interfaz de trabajo continuo.

Cuanto más claro sea ese texto, mejores decisiones tomarán los agentes y menor será el coste de iterar.

Primero funciona.  
Luego se prueba.  
Después se limpia.  
Y para limpiar bien: primero herramientas deterministas locales, después skills.

Empieza mañana por lo mínimo viable: toma una historia pequeña, deja que el agente llegue a "funciona", blíndala con tests, limpia en una segunda pasada y mide cuánto tardas frente a tu flujo habitual.  
Si ganas velocidad sin subir incidencias, ya tienes proceso.
